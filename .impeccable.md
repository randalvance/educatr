# educatr — design & product context

This file is the source of truth for UX, visual, and product decisions. Future
design work must start by reading this.

## Product

educatr is an AI-driven learning companion. The user explores any subject through
conversation; the system distils those conversations into granular, persistent
**Topics** — standalone lessons that fuel downstream artifacts (quizzes today,
flashcards and visual explainers later). It is single-user for the MVP.

Core loop: **chat → topic extraction → dedup/grouping → study artifacts**.

## Design Context

### Users

**Curious adult learners in a calm-focus state.** Self-directed, time-scarce,
reading because they want to — not because they have to. Typical context:
evenings, weekends, quiet coffee shops, between meetings. They are not
gamifying; they are not rushing. They resent infantilising UX, streak badges,
confetti, or any register that assumes they need cheerleading. They *do*
appreciate energy, momentum, and a design that takes the subject matter
seriously.

Job to be done: *"Help me genuinely understand this thing, and make it easy to
come back and consolidate what I've learned."*

### Brand Personality

Three words: **curious, confident, considered.**

Voice / tone:
- A well-read friend, not a professor. Never a chatbot, never a tutor from a
  K-12 app.
- Crisp, specific, assumes intelligence. No filler adverbs, no "Great
  question!" openers, no emoji in product copy.
- Earnest about ideas. Willing to admit uncertainty.

Emotional target: **"I just read something good and I want to keep going."**
Not "I earned a star," not "the AI is sparkling at me."

### Aesthetic Direction

**Bold forms, quiet rhythm.** The user asked for bold & energetic; the audience
asked for calm focus. The resolution is not a compromise — it's strong
typographic contrast, confident negative space, and one saturated accent,
rather than loud color everywhere.

Points of reference (feel, not copy): Anthropic.com's editorial restraint,
Readwise Reader's reading-first chrome, Linear's typographic confidence,
Substack's respect for long-form. Explicit anti-references: ChatGPT's grey
neutrality, Duolingo's reward-loop vocabulary, generic AI-app purple
gradients + sparkle iconography, any "your streak is on fire" register.

**Theme:** Both light and dark, system-driven, with a manual override.
Dark mode is a first-class surface — not a filter.

**Palette — working proposal (reviewable):**
- Ink (text / near-black):  warm `#111110`
- Parchment (light-mode surface):  `#faf8f3`
- Char (dark-mode surface):  `#141311`
- Accent / "Kindling":  burnt orange `#c2410c` — used sparingly, for primary
  actions, live-stream indicators, and moments that should feel alive.
- Warm neutrals:  a stepped grey scale biased toward warm (`#6b6762`-ish), not
  blue-grey.
- Semantic: success `#166534` (evergreen), danger `#b91c1c` (warm red), info
  borrowed from the accent.

Rationale for burnt orange: it carries energy without leaning AI-trope purple
or indigo; it reads as earthy and adult; it holds up on both warm cream and
warm char backgrounds.

**Typography:**
- Display / UI: a confident humanist sans (start with system-ui's San
  Francisco / Segoe UI, graduate to Inter or Söhne when the design budget
  allows). Tight tracking at display sizes.
- Long-form (topic body, chat messages): a readable serif (start with New York
  / Georgia, graduate to Source Serif or Iowan Old Style). Serifs signal
  "this is meant to be read, not skimmed."
- Mono: JetBrains Mono / Menlo — used for inline code and for small
  metadata labels (timestamps, source tags) as a subtle "technical precision"
  cue.

**Motion:**
- Earnable, not decorative. Token-level streaming in chat is the primary
  motion surface; everything else should feel still unless it has a reason.
- 200ms ease-out for surface changes; respect `prefers-reduced-motion` (fall
  back to opacity-only transitions, no transforms).

**Spacing & composition:**
- Generous. Default reading measure ~68 ch. Sidebars narrow (≤ 260–280px).
  Prefer one idea per screen column over dashboard-style density.
- Hierarchical: topic title >> topic summary >> topic body. Users should
  always know what they're looking at in one glance.

### Design Principles

1. **Ideas are the protagonist.** Chrome is thin; the lesson, the question,
   the answer are the loudest things on screen. Navigation and meta recede.
2. **Bold type, quiet color.** Hierarchy comes from typography and
   whitespace, not saturation. One accent colour, used intentionally.
3. **Respect the adult.** No gamification vocabulary, no infantilising
   copy, no celebratory motion. Rewards are intellectual, not visual.
4. **Calm by default, alive on purpose.** The interface sits still until
   something earns motion — a streamed token, a newly extracted Topic, a
   submitted quiz answer.
5. **Dark mode is first-class.** Every colour and shadow decision must work
   in both themes, not one translated into the other.

### Accessibility

- Target **WCAG 2.2 AA** (contrast 4.5:1 body / 3:1 large, visible focus
  rings, keyboard-navigable everywhere).
- Never rely on colour alone to communicate state (pair with icon or copy).
- Honour `prefers-reduced-motion` for all transitions involving movement
  beyond opacity.
- Minimum touch target 44×44 on mobile surfaces (matters for the forthcoming
  React Native app).

### Currently Implemented

Web app `apps/web/` is a minimal scaffold: inline `style={}` only, no CSS
files, no tokens yet. Every design decision above is aspirational until
codified as tokens + components. Any future design pass should start by
extracting the palette and type scale above into actual tokens before adding
UI surface area.
